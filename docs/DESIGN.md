# Repo design

## Goals

1. **Reproducibility.** Anyone can take an upstream chat template, apply our
   patch series, and get a byte-identical patched template. The render
   harness verifies this in CI.
2. **Provenance.** Every upstream file has a recorded SHA-256 + fetch URL +
   fetch date. Every patch has an attribution and a link to the original
   discussion or PR.
3. **Patch granularity.** Each patch addresses one bug. No bundled commits.
   This makes upstreaming easier and reverting safer.
4. **Runtime portability.** Patches are plain text edits to
   `chat_template.jinja` files. Applying them to GGUF embedded templates,
   `model.yaml` overrides, or in-memory tokenizer overrides is the user's
   responsibility — but the source-of-truth diff is the same.

## Layout rationale

```
templates/
  <family>/
    upstream/    ← verbatim. Only touched by scripts/fetch-upstream.sh.
    patched/     ← derivative. Generated by applying patches/ to upstream/.
    PROVENANCE.md ← per-file SHA + URL + date.
patches/
  <family>/
    <id>-<short-name>.patch ← unified diff, applies with `patch -p1`
                              from the repo root.
docs/
  PATCH-CATALOG.md ← every patch with status, attribution, fix detail.
  DESIGN.md         ← this file.
tests/
  fixtures/        ← message-set JSON inputs. One per patch + canonical
                     happy-path samples.
  golden/          ← per-(template, fixture) rendered output. Asserted
                     byte-equal in CI for unchanged templates; intentionally
                     diff for patched templates (see test_render.py).
  test_render.py   ← the harness.
scripts/
  apply.sh         ← copy or symlink a patched template into a model dir.
  verify.py        ← render and diff against goldens; reports drift.
  fetch-upstream.sh ← refresh upstream/ from canonical sources.
```

## Why a flat patch ID space

P1–P10, R1–R5, Q3.6-1, G1–G7 are all global identifiers. There is no
`P1` for both Qwen3.5 and Gemma 4. The cost of unique IDs is small (two
characters); the benefit is unambiguous reference in commit messages,
issue threads, and tests. This matches the convention used in
the upstream `barubary/qwen3.5-barubary-attuned-chat-template` Hugging Face
repo, which numbered its 21 fixes globally.

## What goes in `patched/` vs only in `patches/`

`templates/<family>/patched/<size>.jinja` is a fully-applied derivative
template ready to drop into a model directory. It has the **default patch
stack** for that family applied:

- **Qwen3.5 default stack:** P1 + P2 + P5 + P6–P10 + R1 + R2 + R3
- **Qwen3.6 default stack:** Q3.6-1 only (R1-equivalent fix already
  partially addressed by `preserve_thinking` semantics; G7-equivalent doesn't
  apply since Qwen3.6 has different tool-call structure)
- **Gemma 4 default stack:** G7 only (G1, G4, G5 are opt-in)

If you want a non-default combination, regenerate `patched/` by applying the
patches you want from `patches/<family>/` against `upstream/<size>.jinja`.

The opt-in patches (P4, P5 standalone, G1, G4) are *not* in the default
`patched/` set because they target one specific runtime or workflow. Apply
them ad-hoc.

## Render harness conventions

`tests/test_render.py` uses `jinja2.Environment` directly with a
minimal set of HF-compatible globals/filters:

- `raise_exception(msg)` — raises `RuntimeError(msg)`.
- `tojson` — pre-registered in jinja2.
- `bos_token`, `eos_token`, `pad_token` — passed as variables, default to
  empty strings.

For each `(template, fixture)` pair the harness:

1. Loads the template (upstream or patched).
2. Loads the fixture JSON (`messages`, optional `tools`, optional kwargs,
   `_applies_to` list of family names that filters which templates this
   fixture should render against).
3. Renders.
4. Asserts:
   - **Smoke path:** rendered output is non-empty.
   - **Coverage:** every declared family must render at least one
     applicable fixture — see `test_declared_families_ship_patched_templates`
     and the per-family check inside `test_basic_chat_renders`. This
     prevents an empty `patched/` directory from silently skipping the
     family while the suite stays green.
   - **Bug-fix path:** for the upstream rendering, asserts the bug pattern
     is present (e.g., missing `<turn|>` after empty-content tool call).
     For the patched rendering, asserts the bug pattern is absent.

Bug-fix tests are the load-bearing part of CI — they prevent silent
regressions when re-applying patches against new upstream template revisions.

**`tests/golden/` is intentionally empty in the shipped repo.** Byte-equal
drift detection is an opt-in workflow via `scripts/verify.py --write-goldens`
that captures current renderings as a local baseline. If you want that
coverage, run it once after the initial checkout, commit the captured
goldens to a fork, and use `scripts/verify.py` (without `--write-goldens`)
to detect drift on subsequent template updates. The pytest suite makes no
assertions against `tests/golden/`.

## Adding a new patch

1. **Reproduce the bug.** Capture the failing message set as a fixture
   under `tests/fixtures/`.
2. **Write the failing test.** In `tests/test_render.py`, add an upstream
   render test that asserts the bug pattern is present.
3. **Author the patch.** Edit `templates/<family>/patched/<size>.jinja`
   and capture the diff with
   `diff -u templates/<family>/upstream/<size>.jinja templates/<family>/patched/<size>.jinja > patches/<family>/<id>-<short-name>.patch`.
4. **Write the passing test.** Assert the bug pattern is absent in the
   patched render.
5. **Update the catalog.** New row in `docs/PATCH-CATALOG.md` with status,
   target, prior art, attribution.
6. **Update NOTICE.** If the patch is derived from someone else's work,
   credit them.
7. **Update CHANGELOG.md.** New entry under `[Unreleased]`.

## When upstream merges a patch

Don't delete the catalog entry. Change its status to `upstream` and update
the "Of which already upstream" count in the README's status snapshot.
The patch and tests stay in place as a regression sentinel — if a future
upstream template regression re-introduces the bug, our test catches it.

## Why no `transformers` dependency in the harness

`jinja2` is the lowest common denominator. Most runtimes (HF transformers,
mlx-lm, llama.cpp's Python bindings) wrap a `jinja2.Environment` for chat
template rendering. Testing directly against `jinja2` catches more
ecosystem-relevant bugs than testing through the transformers
`AutoTokenizer` API would, and avoids pulling a 4 GB dependency tree into
CI.

For LM Studio's `minijinja` runtime, the harness has a separate
`tests/test_minijinja.py` skip-if-not-installed test that renders against
the `minijinja` Python bindings (if available). This is how `is sequence`
(G1) and `| safe` (P4) failure modes are caught.
