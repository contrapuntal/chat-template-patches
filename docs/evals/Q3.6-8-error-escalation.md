# Eval: Q3.6-8 — tool-error escalation counter (watch / promotion gate)

**Status:** Q3.6-8 is **watch-listed, NOT implemented.** This doc + the fixture
`tests/fixtures/qwen36_repeated_tool_failures.json` define the bar a candidate
implementation must clear before it may ship (even as opt-in). See
`docs/PATCH-CATALOG.md` § Q3.6-8.

## What Q3.6-8 would do

froggeric's `Qwen-Fixed-Chat-Templates` (v15/v16) adds a pure-Jinja forward
counter: `ns_flags.last_tool_failed` / `consecutive_failures`. On the **first**
detected tool failure it seeds a `<think>` correction directive; on the
**2nd+ consecutive** failure it emits an out-of-band directive. Detection is a
**substring heuristic**: the tool response (when < 500 chars) is scanned for
`error` / `fail` / `invalid` / `exceeds` / `retry` / `exception`.

## Why it is gated (the risks)

1. **False positives.** The substring heuristic fires on legitimate data that
   merely *contains* a trigger word ("0 errors found", `/var/log/error.log`,
   "no exception thrown"). froggeric kept bolting on ad-hoc guards (`'$ '`,
   `'took '`) across versions — a sign the heuristic is not stable.
2. **Prefix/KV-cache symmetry.** Q3.6-8 mutates the assistant generation-prefix
   based on *prior tool outcomes*. The same assistant turn must serialize
   identically as history vs. when first generated, or every follow-up turn
   misses the prefix cache — the exact failure class R1 / Q3.6-2 fix. Any
   directive injected into history must be reproduced byte-identically when
   that turn is later re-rendered.
3. **Unverifiable directive text.** Like Q3.6-7, the injected directive is
   model-facing prompt engineering whose *efficacy* a render-diff cannot prove.

## Acceptance criteria (must pass ALL before promotion)

Given a candidate Q3.6-8 patched template, render each `_eval_cases` entry in
the fixture and check the directive signal (`_escalation_signal`):

| Requirement | Cases | Pass condition |
|---|---|---|
| **True positive — tier 2** | `tp_two_consecutive_genuine_failures` | directive present (escalated) |
| **True positive — tier 1** | `tp_first_failure_only` | directive present |
| **Counter resets on success** | `reset_after_success` | tier-1 (not tier-2) — a success between failures must reset the count |
| **No false positive (data)** | `fp_legitimate_error_count_data` | **NO** directive |
| **No false positive (path/id)** | `fp_identifier_and_path` | **NO** directive |
| **No false positive (negation)** | `fp_negated_keyword` | **NO** directive |
| **Prefix symmetry** | all TP cases | the directive injected into a history turn renders byte-identically when that turn is re-rendered as history in a longer conversation (no cache-busting drift) |

The three `fp_*` cases are the hard gate: froggeric's substring heuristic is
expected to FAIL at least one of them. Q3.6-8 may only be promoted if a
candidate passes all three (e.g. by restricting detection to structured failure
signals rather than free-text substrings) **and** demonstrates prefix symmetry.

## Running the eval

The scaffold test `test_q36_8_eval_*` in `tests/test_render.py` self-activates:
it skips while no `qwen3.6` patched template contains `consecutive_failures`,
and runs the matrix above the moment a candidate does. The candidate template
must emit the literal `_escalation_signal` substring wherever it injects a
directive (or update the fixture's signal). A behavioral/accuracy eval on a
live model is still recommended on top of these mechanical checks.

## froggeric v18 supplies the structural detection (2026-06-11 update)

The original watch verdict turned on the **false-positive risk** of v15/v16's
broad substring matching. froggeric **v18** (2026-05-16) directly addresses it:
it shifted detection from substrings to **structural signals** read from short
tool responses — `"error":`, `exception:`, `traceback`, `command not found`,
`invalid syntax`, `failed to` (snapshot:
`docs/sources/hf-snapshots/froggeric-Qwen-Fixed-Chat-Templates-v18.jinja`
line ~235; v20 keeps it). This is the detection strategy the `fp_*` gate cases
were written to force.

What this changes for the gate:
- The three **false-positive** cases (`fp_legitimate_error_count_data`,
  `fp_identifier_and_path`, `fp_negated_keyword`) should be **re-checked
  against v18's structural matcher** — it likely passes the JSON-key/Traceback
  cases, though `"failed to"` and bare `error:` are still semi-broad, so
  `fp_negated_keyword` ("no exception thrown" — note: no `exception:` colon) and
  path-like data deserve a fresh look.
- The **prefix/KV-cache symmetry** requirement is **unchanged and still open** —
  v18/v20 inject the directive into the generation prefix / `tool_response`
  block based on prior outcomes, which is exactly the mutation the gate flags.
  Structural detection fixes the FP axis, not the symmetry axis.

Net: v18 is a credible basis for a future Q3.6-8 implementation, but promotion
still requires passing the symmetry check (and a live behavioral eval).
